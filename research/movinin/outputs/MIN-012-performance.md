# MIN-012 — Movinin scalability and performance assessment

Honest scoring of Movinin's production-readiness across 10 dimensions, plus concrete bottleneck predictions for a target scale of **1,000 hosts × ~100 bookings/day** (~100k checkout sessions/day, ~3,000 active concurrent users at peak).

Scoring is conservative — production-grade requires evidence, not absence-of-bug.

| # | Dimension | Score | Evidence | Recommended fix |
|---|-----------|-------|----------|-----------------|
| 1 | Caching layer | **2/10** | No Redis, no in-memory LRU, no `Cache-Control` strategy. The app actively *disables* HTTP caching: `app.use(nocache())` in `backend/src/app.ts:40`. The only thing called "cached" in the codebase is the Nodemailer transporter (`backend/src/utils/mailHelper.ts:5`). No `package.json` entry for `redis`, `ioredis`, `node-cache`, `lru-cache`. Static `/cdn` is `express.static` (no Cache-Control headers configured). Heavy aggregations (`getFrontendProperties` in `propertyController.ts:838`) recompute on every request. | Add Redis (Upstash or self-hosted) with read-through cache for property search results keyed by filter hash + 60s TTL; set Cache-Control immutable on hashed asset filenames. |
| 2 | Database query patterns | **3/10** | **Index/query mismatch (cross-ref MIN-003 + MIN-009):** the compound index `{agency:1, type:1, rentalTerm:1, available:1, updatedAt:-1, _id:1}` (`backend/src/models/Property.ts:146`) is **partial-fit** for the public search filter set, which is `{agency: $in [...], location: $in [...], type: $in, rentalTerm: $in, available, hidden}` (`propertyController.ts:867-877`) — leading field `agency` matches but `location` is unindexed in that compound, and `hidden` is fully unindexed across all six declared indexes. **N+1 risk:** every search hit performs a `$lookup` against `User`, `Booking` (overlap check), and another `$lookup` against `Location.LocationValue` (commented out but the pattern lives in `bookingController.ts:784`). **Unbounded `find()`:** `Property.find({ agency: id })` with no `.limit()` in `agencyController.ts:152` and `userController.ts:1653` (the agency-deletion path). **In-memory $regex search:** `name: { $regex: keyword, $options: 'i' }` on `propertyController.ts:757` is anchor-less and bypasses the text index that exists at `Property.ts:149`. | Add compound index `{location:1, available:1, hidden:1, type:1, rentalTerm:1, updatedAt:-1}`; rewrite `$regex` keyword path to use `$text` or Atlas Search; paginate the agency-delete fan-out. |
| 3 | Rate limiting | **1/10** | `grep -rn "rate-limit\|express-rate-limit\|rateLimit"` against `backend/src/` returns zero hits. `backend/package.json` has no rate-limit dependency. Auth (`/sign-in`, `/sign-up`, `/forgot-password` in `userRoutes.ts`), checkout (`/create-checkout-session` in `stripeRoutes.ts`), and message-send endpoints are completely unthrottled. `signin` does not even iterate failed attempts. | Add `express-rate-limit` with Redis store; 5 attempts/min on auth, 30/min on checkout per IP+email, 100/min global per IP. |
| 4 | Image handling | **2/10** | Images are stored on the **local filesystem** of the API server: `app.use('/cdn', express.static(env.CDN_ROOT))` with `CDN_ROOT` defaulting to `/var/www/cdn` (`backend/src/config/env.config.ts:249`). Helper auto-creates `/var/www/cdn/properties`, `/var/www/cdn/users`, `/var/www/cdn/locations` on boot (`app.ts:70-75`). No `sharp`, no `imagemin`, no resize/srcset pipeline — `grep` for those returns nothing. Multer accepts up to 50MB via the global `express.json({limit:'50mb'})`. Property gallery returns absolute filenames for the frontend to load full-resolution. No CDN proxy in front (no CloudFront/Cloudflare config in repo). | Move to S3 + CloudFront with `sharp` resizing on upload (200/600/1200/2400 widths); replace `<img src>` with responsive `srcset`. |
| 5 | Background jobs | **1/10** | No queue library at all — `grep` for `bullmq`, `agenda`, `bee-queue`, `node-cron`, `cron`, `setInterval`, queue, in `backend/src/` returns nothing. Email sending is **synchronous** inside request handlers (`bookingController.ts:280` calls `notify()` then awaits SMTP before responding). Push notifications (Expo) ditto. The only "background" mechanism is MongoDB's TTL index on `Booking.expireAt` (`Booking.ts:82`) — that's not a job system, it's a deletion sweeper that runs every 60 s in MongoDB itself. | Introduce BullMQ (Redis) for emails/push/Stripe retries; degrade requests to "fire-and-forget" by enqueuing. |
| 6 | Frontend bundle size & code splitting | **6/10** | Route-level lazy-loading IS done — `App.tsx:17-35` lazy-imports every page component including `Search`, `Property`, `Checkout`. Vite config (`frontend/vite.config.ts:96-119`) defines manual chunk groups for `react`/`react-dom` and `react-router`. **However**: dependencies include the full `@mui/material@7`, `@mui/icons-material@7`, `@mui/x-data-grid@8`, `@mui/x-date-pickers@8`, `leaflet`, `react-leaflet`, `slick-carousel`, `react-slick`, `@paypal/react-paypal-js`, `@stripe/react-stripe-js` — even with code splitting and tree-shake, MUI alone routinely ships 300–500 kB gzip per page that uses non-trivial widgets. No bundle-size budget, no `vite-plugin-bundle-analyzer`, no preact compat. `terser` with 3 passes is good, `minify:'terser'` good. | Add bundle-analyzer; lazy-load Leaflet only on `/search` map view; replace `react-slick` with native CSS scroll-snap; per-icon imports for `@mui/icons-material`. |
| 7 | Webhook reliability | **1/10** | **There is no Stripe webhook endpoint at all.** `stripeRoutes.ts` exposes only three routes: `createCheckoutSession`, `checkCheckoutSession`, `createPaymentIntent` (`stripeRoutes.ts:7-9`). `grep -rn webhook` against `backend/src/` returns zero hits. The "payment confirmation" path is a polling-on-return-URL pattern: Stripe redirects to `/checkout-session/{CHECKOUT_SESSION_ID}` (`stripeController.ts:69`) and the frontend then POSTs the session ID back to `checkCheckoutSession` (`stripeController.ts:98-188`), which retrieves the session, marks the booking `Paid`, and sends emails. **If the user closes the tab between Stripe `success` and the frontend POST, the booking silently expires via the TTL index and the payment is captured but never reconciled.** No retry, no out-of-band reconciliation. The same is true of PayPal. | Add `/api/stripe/webhook` with signature verification, idempotency table, and `checkout.session.completed` handler; treat the return-URL path as a UX hint only. |
| 8 | Concurrency / race conditions | **2/10** | Confirmed in MIN-007: there is **no advisory lock, no transaction, no version field** around the property-availability check. The overlap detection in `getFrontendProperties` (`propertyController.ts:940-989`) reads `Booking` overlaps as part of the search aggregation — but at booking-creation time (`bookingController.ts` create flow → `stripeController.createCheckoutSession`) the overlap check is not re-validated inside a transaction, and the booking row is inserted with `expireAt` set, then "promoted" by `checkCheckoutSession`. Two concurrent renters can each create a `Pending` booking for the same property + dates and both reach Stripe checkout. First one to return wins; second one has paid Stripe but the booking will reconcile to a duplicate `Paid` row. There is also no MongoDB transaction wrapping the multi-document update in `checkCheckoutSession` (`stripeController.ts:132-176`) — between `booking.save()` and the email sends, partial failure leaves the booking marked `Paid` but no notification sent. `updateStatus` does a bulk `$set` without checking current state (`bookingController.ts:501-505`). | Wrap booking-creation in a Mongo transaction with a `findOneAndUpdate` overlap-guard (insert into a `holds` collection with unique index on `(property, overlapping_window)`); add optimistic-concurrency `version` field. |
| 9 | Multi-tenancy isolation | **3/10** | App-level only, with leak risks (cross-ref MIN-005). Auth middleware (`authJwt.ts`) attaches `req.user._id` and `req.user.type` but enforces nothing scope-wise. `bookingController.deleteBookings` does an explicit per-booking ownership check (`bookingController.ts:543`) — good — but `updateStatus` (`bookingController.ts:496`) **does not check that the bookings being mutated belong to the agency making the request**. An agency calling `POST /update-status` with arbitrary booking IDs will mutate them. No row-level security, no Postgres-grade defense in depth. The `Property.find({agency:id})` cleanup path in `agencyController.ts:152` trusts that `id` came from a verified admin route — it does (route guards), but a bug in any new route bypasses it. | Add an agency-scoped query helper that all controllers MUST go through; add ownership assertion to `updateStatus`; consider migrating ACL to a policy library (CASL) so it's enforceable in tests. |
| 10 | Observability | **5/10** | **Sentry is wired** for backend (`backend/src/monitoring/instrument.ts`) with traces sample rate config and `setupExpressErrorHandler` (`app.ts:64-66`). **Winston is wired** (`utils/logger.ts`, dependency `winston@3.19`). **Missing:** no Prometheus/Datadog metrics, no APM agent, no per-request structured log context (no request-ID middleware, no AsyncLocalStorage), no log-correlation between front+back, no health/readiness endpoints (no `/healthz`, `/readyz`), no DB slow-query log, no business metrics (booking conversion, payment failure rate). Sentry traces sample rate defaults configurable but no breadcrumbs from controllers. | Add `pino-http` with reqId middleware, expose `/metrics` for Prometheus, add `/healthz` for the load-balancer, ship business metrics to Sentry as `Sentry.metrics.increment()`. |

**Average: 2.6 / 10 — well below production-grade for a multi-tenant booking marketplace.**

## Bottleneck predictions at scale (1,000 hosts × 100 bookings/day)

Concrete failure modes ranked by how soon they bite, each pinned to a file path or schema observation.

1. **MongoDB falls back to in-memory sort under sustained search load.** The 6-field compound index at `Property.ts:146` leads with `agency`, but the actual public search filter (`propertyController.ts:867-877`) uses `agency: { $in: [...] }` (1,000 ObjectIds for "all agencies") + `location: { $in: locationIds }` + `available + hidden`. With `$in` on the leading field over 1,000 values and `hidden` not in any index, the planner does an index-scan-then-fetch-then-filter, then an in-memory sort by `updatedAt`. At ~200 search QPS (peak shopping hours) the sort buffer alone exceeds the 32 MB limit and queries error out or spill to disk. **First user-visible symptom:** intermittent 500s on `/api/frontend/properties` during peak; search latency p95 climbs from 80 ms to 4 s.

2. **Abandoned-checkout TTL deletes spike write load.** `Booking.expireAt` TTL index (`Booking.ts:82`, default `BOOKING_EXPIRE_AT` 60–120 s) means every Stripe checkout creates a Booking row that is auto-deleted if the user doesn't return. At 100k checkouts/day with the conservative ~70% abandonment rate observed for inline checkouts, ~70k TTL deletes/day = ~50/min average, ~500/min during traffic peaks. MongoDB TTL sweeper runs every 60 s on the primary and replicates the deletes — combined with the missing webhook (#7 above), every successful payment that Stripe captures but the user never returns from is **silently lost** to the TTL deletion. Estimated: at 100k/day ~0.5–1% (= 500–1,000/day) of payments orphan.

3. **Local filesystem image storage blocks horizontal scaling.** `express.static('/var/www/cdn', ...)` in `app.ts:51` plus the multer-to-disk uploads in `propertyController` mean *the API server is stateful*. Spinning up a second node behind a load balancer immediately desyncs uploads (a property uploaded on node A is 404 on node B). Disk fill is also a real risk: at 1,000 hosts × ~10 properties × ~8 photos × ~5 MB unresized = ~400 GB on a single VM. Migration to S3 is a forced precondition for any horizontal scaling.

4. **Agency-deletion fan-out is O(n) and unbounded.** `agencyController.ts:152` runs `Property.find({agency:id})` (no limit), iterates each property, and synchronously `unlink()`s each `image` and every entry of `images[]` from disk. For a host with 1,000 properties × 8 images = 8,000 sequential `await fs.unlink` calls inside the request handler. At 1,000-property scale this single admin action will time out (Express default 2 min) and leave the database half-deleted with orphan images. Same shape exists in `userController.ts:1653`.

5. **Email/push send blocks the request thread.** `checkCheckoutSession` (`stripeController.ts:132-176`) awaits `bookingController.confirm()` (sends renter email) → `bookingController.notify()` (sends agency email) → another `notify()` to admin, *all serially, all inside the HTTP handler*, before the 200 response goes back to the frontend that's spinning a "confirming payment" spinner. SMTP latency under load is typically 1–3 s. At 100 bookings/day this is fine; at 1,000 (peak day) the express handler thread pool saturates and response times degrade non-linearly. Combined with #7, an SMTP timeout here leaves a paid booking with no confirmation email and no retry.

6. **Booking aggregation does 4 sequential `$lookup`s with no caching.** The admin `getBookings` aggregation (`bookingController.ts:784-870`) does `$lookup` against `User` (agency), `Location` + nested `LocationValue`, `Property`, `User` (renter) — for *every* page of the booking list, every time it's loaded, with `nocache()` middleware actively forbidding the browser from caching. At 100 bookings/day per host × 1,000 hosts × ~5 admin page-loads/host/day = 5,000 of these aggregations daily, each scanning the unindexed nested locale-filtered `LocationValue`. Will not fail at 1k hosts but will dominate database CPU.

7. **No rate limit anywhere = trivially DoS-able.** `signin` has no throttle (`userRoutes.ts:17`). A single attacker can run unlimited password-spray against any known email; combined with synchronous bcrypt at default rounds, ~50 concurrent signin attempts saturate one node's CPU. At 1,000-host scale the platform is on the open internet and someone *will* try this.

## Fitness verdict for Lctnships

**Adopting Movinin as a fundament would *hurt* lctnships' scalability and security posture, not help it.** The Movinin codebase is shaped like a 2018-era MEAN-stack reference app: monolithic Express + Mongoose, no queue, no cache, no webhook, no advisory lock, app-level multi-tenancy, filesystem image storage. Lctnships today already has, per `CLAUDE.md`: idempotent Stripe webhook (`processed_webhook_events` unique-insert short-circuit), atomic cancel flow with `refund_application_fee:true`, Postgres column-level grants on `users` and `bookings`, RLS, advisory-lock-shaped overlap protection via the booking creation price recalc + DB-side checks, Upstash Redis rate limiting in middleware, Vercel cron with `CRON_SECRET` Bearer auth, Resend with React Email templates, Sentry-equivalent observability. Each of those is a feature Movinin would *regress* to a missing primitive. Movinin's mobile app (Expo) and locale system are nice, but the bones underneath them are weaker than what's already shipped at lctnships.com. Recommendation: cherry-pick the Expo mobile shell and the Leaflet-based map UX as references, **do not adopt the backend** — at lctnships' current scale (~hundreds of bookings/month) you would lose webhook reliability, payment idempotency, RLS, and rate limiting on day one of a swap, and gain nothing back except a less-mature feature set.
